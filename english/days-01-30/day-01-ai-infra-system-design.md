## Day 1: Introduction to AI Infrastructure & LLM Serving Basics

### 1) Topic and Core Examination Areas
- **Topic**: Introduction to AI Infrastructure and Large Language Model (LLM) Serving Basics.
- **Core Examination Areas**: Understanding the AI infrastructure stack, LLM serving fundamentals, request lifecycle, and basic performance metrics.

### 2) Requirement Clarification and Metric Definitions
- **QPS (Queries Per Second)**: Target QPS = 1,000 requests/sec for a mid-sized LLM serving platform.
- **Latency Metrics**: 
  - **TTFT (Time To First Token)**: Target < 200ms for interactive chat.
  - **TP99 Latency**: Target < 2 seconds for full response generation.
- **Throughput (TPS - Tokens Per Second)**: Target system-wide throughput of 50,000 tokens/sec.
- **Memory Size (HBM)**: GPU memory capacity, e.g., 80GB HBM3 per GPU node.

### 3) Core Architecture/Technical Component Design
- **API Gateway**: Receives user requests, performs authentication, and routes to serving engines.
- **LLM Serving Engine**: Core inference engine (e.g., vLLM, TGI, TensorRT-LLM) that manages model loading, batching, and execution.
- **Model Registry**: Stores model versions, metadata, and deployment configurations.
- **Monitoring & Observability**: Collects metrics (latency, throughput, GPU utilization) and logs.

### 4) Deep Dive into Key Technologies and Possible Solutions
- **Request Lifecycle**: Client request -> API Gateway -> Request Queue -> Scheduling/Batching -> Model Execution -> Token Streaming -> Response.
- **Serving Approaches**: 
  - *Framework-native serving* (e.g., HuggingFace TGI, vLLM) vs. *Custom serving engines* (e.g., Triton Inference Server).
  - Comparative analysis: Framework-native offers optimized kernels and easier deployment; custom engines offer maximum flexibility but higher development overhead.

### 5) Trade-off analysis
- **Flexibility vs. Performance**: Custom serving engines provide flexibility but may lack the highly optimized kernels of framework-native solutions like vLLM or TensorRT-LLM.
- **Ease of Deployment vs. Feature Richness**: TGI is easy to deploy via Docker but vLLM offers advanced features like PagedAttention and continuous batching out-of-the-box.

### 6) How to determine the optimal solution
- Choose framework-native solutions (vLLM, TGI, TensorRT-LLM) for standard LLM serving due to their optimized kernels and active communities. Select custom engines only if specific non-standard routing or model architectures are required.

### 7) Glossary: Full names and explanations of nouns and abbreviations
- **AI Infrastructure**: The hardware and software stack (compute, network, storage, orchestration) supporting AI model training and inference.
- **LLM (Large Language Model)**: A deep learning model trained on vast text corpora to understand and generate human-like text.
- **QPS (Queries Per Second)**: A metric measuring the number of requests or queries a system can handle per second.
- **TTFT (Time To First Token)**: The latency from sending a request to receiving the first generated token.
- **TP99 Latency**: The latency threshold below which 99% of all requests complete.
- **TPS (Tokens Per Second)**: Throughput metric measuring the number of tokens generated by the model per second.
- **HBM (High Bandwidth Memory)**: A type of computer memory used in GPUs and AI accelerators, offering very high bandwidth compared to standard GDDR memory.
